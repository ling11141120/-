# 核心工作协议

## Plan 确认流程

Program 创建后、执行开发/分析前，**必须**先进入 Plan 模式与用户对话。Agent 根据 `PROGRAM.md` 的 `type` 字段选择对应清单。

### 对话确认清单 — 开发类（type: development）

| 序号 | 确认项 | 说明 |
|------|--------|------|
| 1 | 需求理解 | 用一句话复述用户需求，确认理解一致 |
| 2 | 目标层级 | 新表/修改表属于哪个分层（ods/dwd/dws/ads/dim） |
| 3 | 业务域 | 归属于哪个业务域 |
| 4 | 表命名 | 按命名规范生成候选表名，请用户确认 |
| 5 | 上游表 | 数据来源是哪些表，是否已有 DDL 可参考 |
| 6 | 字段清单 | 新增/修改哪些字段，每个字段的来源和计算口径 |
| 7 | 粒度 & 周期 | 日/小时/全量/增量 |
| 8 | 口径变更 | 是否有旧口径改新口径（如有，必须记录新旧差异） |
| 9 | 回刷需求 | 是否需要回刷历史数据 |
| 10 | 下游影响 | 本次修改是否影响已有下游表 |

### 对话确认清单 — 分析类（type: analysis）

| 序号 | 确认项 | 说明 |
|------|--------|------|
| 1 | 分析目标 | 用一句话复述，明确要回答什么问题或达成什么结论 |
| 2 | 分析范围 | 哪些 schema / 表 / 对象，包含/排除条件是什么 |
| 3 | 评估框架 | 使用什么标准/规则/维度进行评判（引用 SHARED/knowledge/ 中的规则文件；如规则尚不存在，先在 Plan 阶段结构化定义） |
| 4 | 数据采集 | 需要哪些数据（表结构、DML、运行时指标等），采集方式是什么 |
| 5 | 分析粒度 | 逐表？按业务域？按分层？跨表对比？ |
| 6 | 产出物 | 交付什么、格式是什么（报告/清单/标注/血缘图/配置变更） |
| 7 | 小样验证 | 选取哪几个对象做小样试跑，验证标准是什么 |
| 8 | 验证标准 | 怎么判断分析完成且结论可靠（覆盖率、一致性、可复现性） |
| 9 | 后续动作 | 分析结果是否需要落盘（更新文档/配置/CHANGELOG），是否触发后续 Program |

### 确认后落盘

对话达成一致后，将以下内容写入 Program 目录：

- `PROGRAM.md`：任务目标、背景、方案概要（含 `type` 字段）
- `SCOPE.yml`：写入范围（开发类：DDL/DML 路径；分析类：workspace/scripts/documentation）
- `STATUS.yml`：初始状态（task 列表按类型预置）
- `workspace/PLAN.md`：详细方案（使用 PLAN-TEMPLATE.md 结构，按类型选取章节）

用户确认 PROGRAM.md 和 SCOPE.yml 内容无误后，方可开始执行。

## Scope 控制

- **只修改** SCOPE.yml 中 `write` 允许的路径
- **超出范围**时询问用户，不要擅自修改
- `forbidden` 列表中的路径绝对不能写入
- 分析类 Program 默认**不允许修改 `starrocks/` 下的 SQL 文件**，除非 SCOPE.yml 明确授权

## 开发类执行原则

### 改表优先用 ALTER

- 生产表**禁止** DROP + CREATE（会丢数据）
- 新增字段用 `ALTER TABLE ... ADD COLUMN ... AFTER ...`
- 只有全新建表才用 CREATE TABLE

### DML 修改原则

- 改 DML 前**必须先看 DDL**，确认目标表结构
- CTE 名称和输出列名保持稳定，避免破坏下游引用
- 修改 join 逻辑时检查是否会导致 fan-out（一对多膨胀）
- 新增字段在 INSERT 语句中的位置必须与 DDL 字段顺序一致

### 数据口径变更

- 口径变更（改字段来源/计算方式）必须在 Plan 中明确记录
- 记录新旧口径差异、影响范围、是否需要回刷
- 变更需要用户确认后才能执行

## 分析类执行原则

### 规则驱动

- 分析必须有明确的评判标准，不能凭"感觉"
- 如果用户给的规则不够具体，先帮助细化再执行
- 规则边界情况（标准覆盖不到的）必须标记并上报

### 可追溯

- 每个结论标注来源：基于哪个规则文件的哪一条、基于哪段代码、基于哪个查询结果
- 分析报告中的结论格式：`结论 + 依据 + 置信度`

### 增量交付

- 大范围分析不要一次性跑完再汇报
- 先做小样本验证（3-5 个对象），确认分析框架合理后再批处理
- 分批汇报中间结果，给用户提供纠偏机会

### 中间产物管理

分析过程中产生的临时数据/中间结论统一写入 `workspace/` 下的子目录：

| 子目录 | 存放内容 |
|--------|----------|
| `workspace/data/` | 采集的原始数据（元数据查询结果、表结构快照等） |
| `workspace/analysis/` | 逐对象分析结论（中间产物） |
| `workspace/trace/` | 血缘追踪中间结果（探查类专用） |
| `workspace/output/` | 结构化输出（CSV/JSON，供后续处理） |

最终交付物：
- `workspace/report.md`：分析报告（面向人阅读）
- `workspace/RESULT.md`：最终成果总结（Program 完成时写）

## 状态持久化

工作过程中维护以下文件（都在 Program 的 `workspace/` 下）：

| 文件 | 时机 | 内容 |
|------|------|------|
| `STATUS.yml` | 持续更新 | 阶段进度、任务状态 |
| `CHECKPOINT.md` | 上下文紧张时 | 当前状态快照，便于恢复 |
| `HANDOFF.md` | 会话结束未完成时 | 下次继续需要知道的信息 |
| `RESULT.md` | Program 完成时 | 最终成果总结 |
| `validation.md` | 完成验证时 | 验证结果（开发类：validation.sql；分析类：validation.md） |

### HANDOFF 格式

```markdown
## HANDOFF

### 当前状态
- 已完成: xxx
- 进行中: yyy

### 分支
`dev+{owner}+{jira}+{desc}` @ <commit-sha>

### 产出/修改文件
- `{path}` — 改了什么/产出了什么

### 关键决策
- 为什么选了方案 A 而非 B
- 边界情况的处理方式

### 下一步
1. xxx
2. yyy

### 口径变更记录（开发类）
- {字段名}: 旧={来源A} → 新={来源B}
```

### CHECKPOINT 格式

当上下文窗口紧张时，写入 CHECKPOINT.md 保存当前工作快照：

```markdown
## CHECKPOINT

### 目标
当前 Program 的目标（一句话）

### 已完成
- [x] 任务 1
- [x] 任务 2

### 进行中
- [ ] 任务 3（进度：60%，卡在 xxx）

### 关键决策
- 选择方案 A 因为 yyy

### 文件变更/产出
- `{path}` — 改了什么/产出了什么
```

## 上下文管理

Agent 的上下文窗口有限。大任务必然跨越多次会话，HANDOFF / CHECKPOINT 机制保证跨会话零信息损失。

### 主动保存（推荐）

当对话已经很长、接近上下文上限时，用户会要求你保存进度：

1. commit 当前分支的所有变更
2. 更新 **STATUS.yml**（任务进度）
3. 写入 **workspace/HANDOFF.md**（交接文档）
4. 如果上下文特别紧张，额外写 **workspace/CHECKPOINT.md**
5. 用户开新会话，说"继续 P-YYYY-NNN"，Agent 从 HANDOFF / CHECKPOINT 恢复

### 文件通信原则

- 大段 SQL、分析报告、采集数据写入文件，不要塞进对话
- 引用文件路径而非复制内容
- 保护上下文窗口
- 分析类任务尤其注意：批量分析对象的中间结论不要逐条在对话中展示，汇总后写入文件

## 工作节奏

### 开发类

1. Plan 确认 — 与用户对话，完成开发类 Plan 确认清单
2. 落盘 — 写入 PROGRAM.md、SCOPE.yml、STATUS.yml、PLAN.md
3. 编码 — 按 DDL → DML 顺序开发
4. 验证 — 编写验证 SQL 检查数据质量
5. 更新 — 完成后更新 STATUS.yml

### 分析类

1. Plan 确认 — 与用户对话，完成分析类 Plan 确认清单
2. 落盘 — 写入 PROGRAM.md、SCOPE.yml、STATUS.yml、PLAN.md
3. 数据采集 — 扫描仓库 / 查询 SR / 编写辅助脚本
4. 小样验证 — 少量对象试跑，确认框架，用户确认方向
5. 批量执行 — 分批分析，中间结果写文件
6. 汇总 — 生成分析报告和结构化输出
7. 验证 — 按验证标准自检，用户确认
8. 交付 — 更新 STATUS.yml、写 RESULT.md、提交代码
